# Contributing

Спасибо за вклад в `EGE Helper Bot`.

## Перед началом

1. Убедись, что есть Python `3.12+`.
2. Убедись, что запущен Docker Desktop.
3. Создай `.env` из шаблона:
```bash
cp .env.example .env
```

## Запуск в Docker

1. Подними весь стек (бот + PostgreSQL + Redis):
```bash
docker compose up -d --build
```

2. Проверь, что сервисы healthy:
```bash
docker compose ps
```

3. Посмотри логи бота:
```bash
docker compose logs -f bot
```

## Локальный запуск бота (Python на хосте)

1. Подними только БД и Redis:
```bash
docker compose up -d postgres redis
```

2. Создай и активируй виртуальное окружение:
```bash
python3 -m venv .venv
source .venv/bin/activate
```

3. Установи зависимости:
```bash
pip install -U pip
pip install -e .
```

4. Запусти бота:
```bash
python -m bot.main
```

## Git workflow

1. Создавай отдельную ветку от `main`:
```bash
git checkout -b feature/short-description
```

2. Делай небольшие логичные коммиты.
3. В названии ветки используй префиксы:
- `feature/...` для новых фич
- `fix/...` для исправлений
- `chore/...` для инфраструктурных изменений
- `docs/...` для документации

## Стиль кода

- Следуй PEP 8.
- Пиши понятные имена переменных, функций и модулей.
- Пиши короткие комментарии только там, где без них сложнее понять контекст.
- Не смешивай несвязанные изменения в одном PR.

## Работа с конфигами

- Не коммить секреты и реальные токены.
- Не меняй `.env.example` без необходимости.
- Если добавляешь новые переменные, обнови:
1. `.env.example`
2. `src/bot/config.py`
3. `README.md`

## Проверки перед PR

Перед созданием PR проверь минимум:

```bash
python -m compileall src
docker compose config
```

Если добавлены тесты, запусти их локально и приложи результат в описание PR.

## Pull Request checklist

В описании PR укажи:

1. Что изменено.
2. Зачем это изменение нужно.
3. Как это проверить локально (конкретные шаги).
4. Что может сломаться (если есть риски).

## Что не принимается

- Коммиты с реальными токенами и паролями.
- Большие PR без объяснения мотивации.
- Изменения без базовой локальной проверки.

## Вопросы и предложения

Если не уверен в архитектурном решении, сначала открой issue с коротким RFC-описанием:

1. Проблема
2. Предложение
3. Альтернативы
4. Риски
